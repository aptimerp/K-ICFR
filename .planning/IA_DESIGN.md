## 0. 시스템 정의 (v4 신규)

### 한 줄 정의
> Smart-ICM은 **비개발자인 내부회계팀이 자유롭게 유지보수할 수 있는 K-ICFR 평가 운영플랫폼**이다.

### 7가지 집중 기능
| # | 기능 | 의미 |
|---|------|------|
| F1 | **평가 Workflow 관리** | Status 8단계 흐름 자체. 누가 무엇을 다음에 해야 하는지 명확 |
| F2 | **샘플 관리** | 모집단에서 평가 대상 샘플 선정·배정·추적 (P5 본업) |
| F3 | **증빙 수집** | ERP 자동 수집 + 수동 업로드. 표준 양식·필수 항목 강제 |
| F4 | **평가 상태 관리** | 통제·샘플별 현재 Status, 다음 액션 책임자, 이전 이력 |
| F5 | **감사 패키지** | 운영평가 결과를 트리 구조 ZIP으로 자동 생성, 감사 제출용 |
| F6 | **AI 검증** | 증빙 품질·누락·정합성 검증 + 평가 초안 생성 (보조 역할) |
| F7 | **이력 관리** | 누가·언제·무엇을 변경했는지 전수 추적. 감사 대응 핵심 |

### 비개발자 친화 원칙
- **노코드 설정** — RCM·통제·증빙 마스터를 화면에서 직접 편집 (DB 직접 접근 불필요)
- **위저드 가이드** — 평가 시작·종료·감사 패키지 생성 등 핵심 작업은 단계별 안내
- **이력 추적 자동** — 모든 변경은 자동 기록, 별도 관리 행위 불필요
- **시각화 우선** — 진행률·상태·이상 탐지는 그래픽으로, 표는 보조
- **AI는 도우미** — 사람이 최종 판단, AI 결과는 항상 검토 가능한 형태로

### 명시적 비범위 (Smart-ICM에서 하지 않는 일)
- ❌ 전표 생성·결재 (ERP·그룹웨어)
- ❌ 증빙 원천 저장소 (그룹웨어)
- ❌ 인사정보 마스터 (HR)
- ❌ 권한 SSO 마스터 (SAC 합의 후 연계)
- ❌ 설계평가 (Phase 6에서 구축)
- ❌ **RCM·Scoping의 무거운 시스템 관리** (v5: 엑셀 템플릿 기반 가벼운 운영)

### 평가 주기·모드 (v5 신규 — 시스템 핵심 룰)
회사는 연 1회 설계평가 + 3회 이상 운영평가를 수행한다.

| 평가 모드 | 시기 | 대상 모집단 | 평가 방식 |
|---------|------|-----------|---------|
| **설계평가** | 연 1회 (연초) | RCM 전체 | Phase 6 — 본 구축 범위 밖 |
| **중간 1차** | 1~6월 분 | 해당 기간 모집단 | **샘플링 기반** |
| **중간 2차** | 1~9월(또는 10월)분 | 해당 기간 모집단 | **샘플링 기반** |
| **기말 평가** | 잔여기간~12월 분 | 잔여기간 모집단 | **샘플링 생략**, 부족 증빙 보충 방식 ⭐ |

#### 기말평가 특수 흐름 (v5 신규 · v6 수정)
1. P5가 통제활동별 **모범규준에 따른 부족 증빙 수** 파악
2. 통제별 "필요 증빙 N건" 표시 → **P1이 자율 선택 제출**
   - **제출 우선순위 (v6)**: **12월 31일 기준 최신 증빙 우선** (12월 미발생 시 11월 → 그 이전 순)
3. 기말평가 후 통제별 **모집단 개수 대비 평가 증빙 충족 여부** 자동 체크
4. → 샘플링·부서승인 워크플로우 스킵, P1 자율제출 → P3·P5 검증으로 직행

### 모집단 빈도 규칙 (v5 — ERP 양식 기반)
| 통제 주기 | 9월 기준 모집단 | 비고 |
|---------|--------------|------|
| 분기 | 3개 (3·6·9월) | 연간 4개 |
| 월별 | 9개 (1~9월) | 연간 12개 |
| 일별·발생기준 | 전체 발생건수 | |
| 연간 | 0개 (9월 기준) | **기말평가 시점에만 평가 가능** |

### 연결 내부회계 (v5 신규 — 멀티엔터티) ⚠️ v6 변경: 후속 페이즈 도입
- **현 Phase 5~6 범위 밖** (v6): 단일 엔터티(모기업) 기준으로 우선 구축
- 후속 페이즈에서 멀티엔터티 도입 시 다음 설계 그대로 적용:
  - ERP 양식의 **ENTITY_CODE** (예: 10000 = 모기업) 컬럼 기반
  - Smart-ICM은 모기업 + 관계회사 모두 사용
  - 모기업 P5는 자기 회사 + 관계회사 데이터 종합 조회
  - 관계회사 P5는 자기 회사만
  - 헤더에 회사 셀렉터 (모기업 P5에게만 노출)
  - 연결 평가 보고서 작성 시 모기업이 종합
- DB 스키마 설계 시점에 ENTITY_CODE 컬럼만 **선반영** (향후 확장 비용 ↓)

## 2. 페르소나 (사용자 5종)

### 🟦 P1 — 통제수행자 (1순위, 현업부서원 70~80명 추정) ⭐
- **소속**: 현업부서 (재무회계·구매·영업·생산·인사 등)
- **예**: 김민준 과장 (재무회계팀)
- **사용 빈도**: 주 3~5회 · 1회당 10~30분
- **핵심 동기**: "내가 담당한 통제의 자가평가·증빙제출을 빨리 끝내고 본업으로"
- **주요 작업**:
  1. 본인 담당 통제의 **설계평가 자가평가** 수행
  2. 본인 담당 통제의 **운영평가 증빙 제출** (파일 업로드 + 의견)
  3. 반려된 건 재작성·재제출
- **권한**: 본인 담당 통제만 (다른 통제 데이터는 보이지 않음)

### 🟧 P2 — 통제책임자 (현업부서장, 15~20명)
- **소속**: 현업부서장 (재무팀장·구매팀장 등)
- **예**: 이차장 (재무회계팀장)
- **사용 빈도**: 주 5~10회 · 1회당 5~15분
- **핵심 동기**: "우리 부서원이 올린 자가평가·증빙을 빠르게 검토·승인"
- **주요 작업**:
  1. 부서원(P1)의 자가평가 결과 **검토·승인·반려**
  2. 부서원이 제출한 **증빙 검토**
  3. 부서 차원의 미비점 인지
- **권한**: 본인 부서 전체 (소속 P1들의 데이터)

### 🟪 P3 — 내부회계평가자 (TF 인원, 5~10명) ⭐ — v5 수정
- **소속**: 내부회계평가 TF (재무팀 또는 전사 차원 차출)
- **예**: 장한나 PL (TF 리더)
- **사용 빈도**: 매일 · 1~3시간
- **핵심 동기**: "TF로서 다른 부서를 평가하면서, 내 부서 통제도 동시 수행"
- **주요 작업** (역할 2~3중 수행):
  1. **P1 역할 (담당자 수행)** — 본인 담당 통제의 자가평가·증빙제출
  2. **P2 역할** — (P3가 부서장 겸직 시) 부서원 자가평가 승인
  3. **TF 본업 (평가자 수행)** — 외부 평가자로서 다른 부서 통제의 **운영평가 수행**
     - 증빙 검토 후 **정상 / 예외 / 해당없음** 3분류로 판정
     - **AI 평가 초안 확인** → 사람이 최종 운영평가 의견 작성 (v5 신규)
     - **예외 판정 시 → 예외로 분류한 상세 평가의견 의무 기재 + 미비점 등록** (v5 강화)
- **샘플링은 P3 작업이 아님** — P5가 독립적으로 수행
- **평가 결과 3분류**:
  - 🟢 **정상 (NORMAL)**: 통제활동이 설계된 대로 작동
  - 🟡 **예외 (EXCEPTION)**: 일부 미흡 — **상세 의견 의무 + 미비점 등록**
  - ⚪ **해당없음 (NOT APPLICABLE)**: 평가 대상 거래·사건 부재
- **P3 홈화면 핵심 UX (v5)**:
  ```
  대시보드 영역을 2분할:
  [담당자 수행 (P1 해당)]     ← 본인의 P1 작업 (증빙 제출 등)
  [평가자 수행 (P3 해당)]     ← TF로서 평가 작업 (운영평가 수행)
  ```
  → 같은 사람이 두 가지 다른 모드 작업을 명확히 구분
- **권한**: 본인 담당 + 본인 부서 + TF 평가대상 부서 (샘플링·시스템 설정 제외)

### 🟨 P4 — 감사인·감사위원회·내부회계관리자 (5~10명) — v4 수정
- **소속**: 외부감사 회계법인 / 감사위원회 / 내부회계관리자
- **예**: 외부감사 담당 회계사, 감사위원, 내부회계관리자(임원)
- **사용 빈도**: 월 1~2회 또는 분기말·연말 집중
- **핵심 동기**: "K-ICFR 운영 상태를 객관적으로 확인 + 최종 보고서 수령"
- **주요 작업**:
  1. **RCM 조회** — 통제 정의·증빙 요건 확인
  2. **평가현황 조회** — 운영실태평가 보고서 / 진행률 / 미비점 현황
  3. **통제이력 조회** — 통제별 평가 결과 이력 / 변경 이력 (감사 추적)
  4. 운영실태평가 보고서·이사회 보고서 **PDF 다운로드**, 감사 패키지 **ZIP 다운로드**
- **권한**: **제한된 읽기 전용 (v4 수정)** — RCM / 평가현황 / 통제이력 3개 영역만 조회 가능
- **권한 제외 영역**: 모집단·샘플링 / 증빙 수집 현황 / 내 증빙 제출 / AI 검증 / 부서 승인 / 운영평가 수행 상세 / 시스템 설정 (보안·기밀성)

### 🟥 P5 — 내부회계팀 (관리자, 2~3명) — v5 수정

### P1 통제수행자의 Day-1 시나리오 (가장 자세히) — v3 수정

     ② **반려된 재제출 1건** (긴급, 빨강 배지)
   → (설계평가 카드 없음 — Phase 6)

4. 끝. 본업으로 복귀.

### ⭐ RCM 지정 기능 (v6 핵심 신규)
- **목적**: 운영평가 시작 시점의 RCM 버전을 "이 버전으로 평가했다"고 시스템에 확정 기록 → 감사 추적 키
- **운영 룰**:
  - 1️⃣ **첫 운영평가(중간 1차) 시작 전 — 반드시 RCM 지정 선행**
  - 2️⃣ **기말 운영평가 시작 전 — 필요시 RCM 지정** (설계평가·변화관리로 RCM 변경됐을 때)
- **흐름**:
  ```
  설계평가/변화관리 진행 → RCM 변경사항 누적 → 1️⃣ RCM관리 → RCM 지정 탭에서 엑셀 업로드
  → 현 시점 RCM이 운영평가용 버전으로 락(lock) → 운영평가 시작 가능
  → 운영평가 결과에 "RCM 버전: vN (지정일: YYYY-MM-DD)" 자동 첨부
  ※ 설계평가 진행 여부와 무관하게 RCM 지정만으로 운영평가 시작 가능
  ```
