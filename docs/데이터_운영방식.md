> Phase 8 | 2026-06-01 | IA v7 + DB 스키마 v7 기반
> 목적: 각 탭의 데이터 원천·CRUD 패턴·상태전이·권한을 코딩 전에 확정

## 범례

| 기호 | 의미 |
|------|------|
| ✅ ERP API | SAC 제공 ERP API로 자동 수신 (Phase 9 연동 전까지 Mock 사용) |
| 📥 엑셀 업로드 | 표준 템플릿 → 시스템 일괄 등록 |

## 1️⃣ 운영관리

### 1-1. 평가기간 관리 (`eval_terms`)

**데이터 원천**: Supabase 직접 관리 (ERP 연동 없음)

**CRUD 정의**

| 액션 | 설명 | 권한 | 비고 |
|------|------|------|------|
| Create | 새 평가기간 등록 | P5 | `eval_mode`: INTERIM1·INTERIM2·FINAL 선택 |
| Read | 목록 조회·상세 확인 | P3·P5 | 전체 기간 이력 포함 |
| Update | 기간명·날짜·활성 여부 수정 | P5 | `is_active=true`인 기간 1개만 유지 (업데이트 시 기존 비활성화) |
| Delete | 평가기간 삭제 | P5 | 하위 데이터 있을 경우 삭제 금지 (FK 보호) |

**상태 전이**
```
등록(is_active=false)
  → 활성화(is_active=true)  ← P5 수동 전환, 동시에 기존 활성기간 비활성
  → 마감(is_active=false + end_date 도달)  ← 차수 종료 승인 시
```

**UI 주요 기능**
- 평가기간 목록 테이블 (연도·차수·모드·기간·상태)
- 신규 등록 모달 (평가연도, 차수, 모드, 시작/종료일)
- 활성 기간 전환 버튼 (현재 활성 기간 강조 표시)
- 전기간 참조 버튼 — 직전 기간 설정값 복사

**검증 규칙**
- `start_date < end_date` 필수
- 동일 `term_year` + `eval_mode` 중복 등록 금지
- 활성 기간이 1개 초과 불가

### 1-2. 부서·사원 관리 (`departments`, `users`)

**데이터 원천**: ✅ ERP API (원본 창구) → Supabase 동기화

> ERP DEPT_CODE, EMP_NO가 연결 키. Smart-ICM은 조회·보완만, 원본 수정은 ERP에서.

**CRUD 정의**

| 액션 | 대상 | 권한 | 비고 |
|------|------|------|------|
| Read | 부서 목록·사원 목록 | P5 | ERP에서 최종 동기화된 데이터 표시 |
| Sync (ERP 수신) | departments + users 일괄 갱신 | P5 | "ERP 동기화" 버튼 클릭 → SAC API 호출 |
| Update (보완) | Smart-ICM 전용 필드만 수정 | P5 | ERP 원본 필드 수정 금지 (`dept_code`, `emp_no` 등) |
| Soft Delete | `is_active=false` | P5 | ERP에서 퇴직·조직변경 시 동기화로 자동 반영 |

**ERP 동기화 패턴**
```
P5 "ERP 동기화" 클릭
  → Vercel API Route: GET /api/erp/departments, /api/erp/employees
  → SAC API 호출 (KUMKANG_API_URL, Phase 9 연동 전 Mock)
  → Supabase upsert (dept_code / emp_no 기준)
  → 동기화 결과 리포트 (신규 N건·변경 N건·비활성 N건)
```

**UI 주요 기능**
- 탭 분리: [부서 목록] [사원 목록]
- 부서 목록: 트리 구조 (parent_code 기반) 또는 플랫 테이블 토글
- 사원 목록: 부서 필터 + 페르소나(P1~P5) 필터
- ERP 동기화 버튼 + 마지막 동기화 시각 표시
- Smart-ICM 전용 필드 인라인 편집 (예: persona 배정)

**검증 규칙**
- ERP 원본 컬럼(`dept_code`, `emp_no`, `dept_name`, `emp_name`) → 수정 UI 비활성
- `persona` 배정은 Smart-ICM 전용 (P5가 설정)

**엑셀 템플릿 컬럼**: `상신자_사번` · `법인코드` · `결재자1_사번` · `결재자2_사번(선택)` · `단계수`

### 1-4. RCM 지정 (`rcm_snapshots`)

**데이터 원천**: 📥 엑셀 업로드 → `rcm_snapshots` 버전 락

> 운영평가 직전 RCM 버전을 확정 잠금. 이후 `rcm_controls` 수정이 평가 결과에 영향 없음.

**CRUD 정의**

| 액션 | 설명 | 권한 | 비고 |
|------|------|------|------|
| Create (지정) | 현 RCM 전체 스냅샷 → `snapshot_data(JSONB)` 저장 | P5 | 평가기간 1개당 1회만 가능 |
| Read | 지정된 스냅샷 조회 | P3·P5 | 버전·지정일·지정자 표시 |
| ❌ Update | **수정 불가** (🔒 버전 락) | — | 취소 후 재지정만 허용 |
| Delete (취소) | 스냅샷 삭제 (진행중 평가 없을 때만) | P5 | |

**지정 플로우**
```
P5 [RCM 지정 실행] 클릭
  → 확인 모달: "현재 RCM N건을 평가용으로 확정합니다. 이후 변경 불가."
  → rcm_snapshots INSERT (eval_term_id UNIQUE 보장)
  → 운영평가 메뉴 활성화 (locked_at 기준 체크)
```

**UI 주요 기능**
- 현재 활성 평가기간의 RCM 지정 현황 카드
- 지정 전: "RCM 지정 실행" 버튼 (경고 강조)
- 지정 후: 스냅샷 버전 정보 + 지정일 + 지정자 표시 (수정 불가 배지)
- 이전 지정 이력 테이블

## 3️⃣ 운영평가관리

> 헤더 차수 토글 (중간1·중간2·기말) 상태에 따라 모든 데이터 필터링 적용.

### 3-1. 샘플링작업 — 모집단 관리 (`populations`)

**데이터 원천**: ✅ ERP API (95%) + 📥 엑셀 업로드 (5% 수기 모집단)

**CRUD 정의**

| 액션 | 설명 | 권한 |
|------|------|------|
| Create (ERP 수신) | ERP API → populations 일괄 upsert | P5 |
| Create (엑셀) | 수기 모집단 업로드 (`source='EXCEL_UPLOAD'`) | P5 |
| Read | 통제별·부서별 모집단 목록 | P5 |
| Delete | 수기 업로드 건만 삭제 가능 (ERP 수신 건 삭제 불가) | P5 |
| 📤 Download | 모집단 엑셀 내보내기 | P5 |

**TRANSACTION_ID 규칙**
- ERP 수신: ERP 원본 TRANSACTION_ID 그대로 저장
- 엑셀 업로드: `MANUAL-{ctrl_code}-{YYYYMMDDHHmmss}` 형식 자동 부여

**UI 주요 기능**
- 통제별 모집단 건수 요약 카드
- "ERP 모집단 수신" 버튼 + 마지막 수신 시각
- 수기 모집단 업로드/다운로드
- ERP 수신분 vs 수기 업로드분 색상 구분

---

### 3-2. 샘플링작업 — 샘플링 작업 (`samples`)

**데이터 원천**: `populations` 기반 P5 직접 선정

**CRUD 정의**

| 액션 | 설명 | 권한 |
|------|------|------|
| Create | 샘플 선정 (4가지 메서드) | P5 단독 |
| Read | 샘플 목록·현황 | P5 |
| Update | 샘플 상태 갱신 (자동·수동) | P5·시스템 |
| Delete | 샘플링 취소 (평가 시작 전만) | P5 |

**샘플링 메서드**

| 메서드 | 설명 | 비고 |
|--------|------|------|
| RANDOM | 시스템 자동 랜덤 선정 | 모집단 N건 중 K건 |
| JUDGMENTAL | P5 직접 선택 | 특정 거래 지정 |
| HAPHAZARD | 비구조적 임의 선정 | |
| 수동 (샘플전산여부=N) | P1이 모집단 직접 기재 | 생산프로세스 5~6건 해당 |

**기말평가 분기**
```
eval_mode = 'FINAL' → 샘플링 단계 SKIP
  → P5가 통제별 "부족 증빙 수" 직접 산정
  → samples 레코드 없이 evidence_submissions로 직행
```

**UI 주요 기능**
- 통제별 샘플링 진행 현황 테이블
- 샘플링 메서드 선택 드롭다운
- 샘플 건수 자동 계산 (모범규준 참조)
- 랜덤샘플링 불가 통제 별도 표시 (노란색 배지)

### 3-7. 증빙제출 — 증빙수집 (`evidence_files`)

**데이터 원천**: ✅ ERP API 자동 수집 + P1 수동 업로드

**CRUD 정의**

| 액션 | 설명 | 권한 |
|------|------|------|
| Create (ERP 자동) | TRANSACTION_ID 기반 ERP 파일 자동 pull | 시스템 |
| Create (수동) | P1·P3·SAC 파일 업로드 | P1·P3·P5 |
| Read | 증빙 파일 목록·미리보기 | P1(본인)·P2(부서)·P3·P5 |
| Delete (소프트) | `is_deleted=true` | P1(본인·제출 전)·P5 |

**파일명 규칙**: `{증빙순번}_{모집단내용}.{ext}`
- 자동 변환: 업로드 시 시스템이 파일명 자동 정규화

**Storage 경로**: `evidence-files/{eval_term_id}/{ctrl_code}/{sample_id}/{file_name}`

**AI 검증 자동 실행** (업로드 완료 시 비동기)
```
파일 업로드 완료
  → 🤖 AI 검증 큐 등록
  → evidence_files.ai_verified / ai_score / ai_note 갱신
  → samples.status = 'AUTO_COLLECTED' 또는 'MANUAL_UPLOADED'
```

### 3-8. 증빙제출 — 증빙 AI 검증 (`evidence_files.ai_*`)

**데이터 원천**: `evidence_files` AI 필드 + `samples` 상태

**AI 검증 항목**

| 검증 항목 | 설명 | 결과 |
|----------|------|------|
| 첨부 누락 | 필수 증빙 파일 수 충족 여부 | PASS/FAIL |
| 날짜 일치 | 증빙 문서 날짜 vs TRANSACTION_DATE | PASS/FAIL/WARN |
| 금액 정합성 | 증빙 금액 vs ERP 데이터 | PASS/FAIL/WARN |
| 승인라인 | 결재선 서명·도장 존재 여부 | PASS/FAIL |
| ai_score | 종합 신뢰도 0~100 | 수치 |

**UI 주요 기능**
- 통제별 AI 검증 결과 현황 테이블 (PASS/FAIL/WARN 색상 구분)
- 개별 샘플 AI 검증 상세 패널 (항목별 결과)
- P5 이상 탐지 모니터링 대시보드 (ai_score < 70 건 강조)
- 🤖 수동 재검증 트리거 버튼

**상태 전이**
```
evidence_files 업로드 완료
  → samples.status = 'MANUAL_UPLOADED'
  → 🤖 AI 검증 완료
  → samples.status = 'AI_VALIDATED'
```

### 3-9. 증빙제출 — 증빙제출 수행 (`evidence_submissions`)

**데이터 원천**: `evidence_submissions` (샘플 1건 = 제출 1건)

**CRUD 정의**

| 액션 | 설명 | 권한 |
|------|------|------|
| Create (DRAFT) | 증빙제출 초안 저장 | P1(본인 담당)·P3·SAC |
| Update | 제출의견 작성·파일 추가·수정 | P1(본인)·P3 |
| Submit | DRAFT → SUBMITTED 상신 | P1·P3 |
| Resubmit | REJECTED → 수정 후 재상신 | P1·P3 |
| Read | 본인 담당 건만 조회 | P1(본인) / P2(부서) / P3·P5(전체) |

**상태 전이 (Status Machine)**
```
DRAFT ──[상신]──→ SUBMITTED ──[P2 승인]──→ APPROVED
                      │                        ↓
                      └──[P2 반려]──→ REJECTED → DRAFT (재작성)
                      └──[P2 보완요청]──→ REVISION_REQUESTED → DRAFT
```

**기말평가 분기**
```
eval_mode = 'FINAL'
  → 샘플 없이 P1이 직접 증빙 선택하여 제출
  → submission_note에 "기말: 12/31 기준 최신" 자동 태그
  → 제출 우선순위: 12월 31일 기준 최신 → 11월 → 그 이전
```

**랜덤샘플링 불가(수동) 모드**
```
samples.sampling_method = NULL (샘플전산여부=N)
  → P1 화면에 "수동 제출 모드" 활성
  → TRANSACTION_ID·날짜·내용 직접 기재 가능
  → 모집단 엑셀 첨부 의무 (노란색 강조)
```

**UI 주요 기능**
- 마이페이지 형태: 본인 담당 통제 목록 + 각 통제의 제출 현황
- 좌측: 담당 통제 목록 (상태 배지), 우측: 선택 건 상세
- AI 검증 결과 미리 표시 (제출 전 확인)
- 파일 드래그앤드롭 업로드
- 반려 사유 빨간색 배너 강조

ctrl_kind = 'ITGC'

### 3-11. 운영평가 — 운영평가 수행 (`op_evaluations`)

**데이터 원천**: `op_evaluations` (샘플 1건 = 평가 1건)

**CRUD 정의**

| 액션 | 설명 | 권한 |
|------|------|------|
| Read | 배정된 평가 목록 | P3(배정분)·P5(전체) |
| Create AI Draft | 🤖 AI 평가 초안 자동 생성 → `ai_draft` 저장 | 시스템 |
| Evaluate | result 선택 + eval_note 작성 → IN_PROGRESS | P3·P5 |
| Finalize | P5 최종 확정 → FINALIZED | P5 |

**결과 3분류**

| 값 | 표시 | 의미 |
|----|------|------|
| `PASS` | 🟢 정상 | 통제활동이 설계대로 작동 |
| `EXCEPTION` | 🟡 예외 | 일부 미흡 → 미비점 등록 의무 |
| `NOT_APPLICABLE` | ⚪ 해당없음 | 평가 대상 거래·사건 부재 |

**상태 전이**
```
PENDING (P2 승인 완료 후 활성)
  → 🤖 AI Draft 생성 → samples.status = 'AI_DRAFTED'
  → P3 평가 작성 → IN_PROGRESS
  → P3 제출 → COMPLETED → samples.status = 'EVALUATED'
  → P5 최종 확정 → FINALIZED → AUDIT_PACKAGED 대기
```

**예외 판정 시 의무 처리**
```
result = 'EXCEPTION'
  → eval_note 필수 입력 (빈 칸 제출 차단)
  → deficiencies 레코드 자동 생성 (severity 선택 UI 노출)
  → 미비점 등록 완료 전까지 COMPLETED 전환 불가
```

**UI 주요 기능**
- 마이페이지: 배정된 평가 목록 + 상태 필터
- AI Draft 박스 (상단 회색 배경, "AI 초안 — 최종 판단은 사람이 합니다" 워터마크)
- 결과 3분류 라디오 버튼 (크게 강조)
- 예외 선택 시 → 미비점 등록 인라인 패널 슬라이드 오픈

### 3-12. 운영평가 — 예외보고 수행 (`deficiencies`)

**데이터 원천**: `op_evaluations.result = 'EXCEPTION'` 시 자동 생성

**CRUD 정의**

| 액션 | 설명 | 권한 |
|------|------|------|
| Create (자동) | 예외 판정 시 `deficiencies` 자동 INSERT | 시스템 |
| Update | severity 선택 + 상세 내용 작성 | P3·P5 |
| 개선담당자 보완 | 🔵 개선권고 → 개선담당자 보완 증빙 첨부 | P1(지정자) |
| 관리 전환 | 🔴 미비점 → 미비점 검토로 라우팅 | P5 |

**예외 등급 4분류**

| 값 | 기호 | 의미 | 후속 처리 |
|----|------|------|----------|
| `IMPROVEMENT_SUGGESTION` | 🔵 | 개선권고 | 개선담당자 보완 증빙 첨부 |
| `SIMPLE_DEFICIENCY` | 🟡 | 단순한 미비점 | 예외보고서 작성 + 결재 |
| `SIGNIFICANT_DEFICIENCY` | 🟠 | 유의한 미비점 | 예외보고서 + 미비점 검토 |
| `MATERIAL_WEAKNESS` | 🔴 | 중요한 취약점 | 즉시 P5 에스컬레이션 |

**상태 전이**
```
OPEN (자동 생성)
  → IN_PROGRESS (개선 착수)
  → CLOSED (개선 완료·P5 확인)
  → WAIVED (예외 인정·P5 결재)
```
