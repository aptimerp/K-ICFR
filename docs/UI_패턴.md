# Smart-ICM UI 표준 패턴
> 재사용 컴포넌트와 확정된 UI 패턴 모음. 신규 화면 구현 전 반드시 이 문서를 확인하고 기존 컴포넌트를 우선 사용한다.
> 최종 수정: 2026-06-08

---

## 1. 증빙 미리보기 — `components/EvidenceViewer.jsx` ⭐ (2026-06-08 확정)

### 적용 대상
**증빙을 개별 단위로 확인하는 모든 화면**은 이 컴포넌트를 사용한다:
- 결재관리 > 상신함·결재함 (적용 완료)
- 운영평가관리 > 증빙제출 수행·승인, 운영평가 수행
- 결과조회 > 평가증빙제출(ZIP)
- 설계평가관리 자체평가 (향후)

### 왜 이 패턴인가 (A4 증빙 가독성 ↔ 다건 비교의 양립)
A4 규격 증빙은 좁은 칸에 넣으면 글씨가 작아 못 읽고, 한 건만 크게 보면 비교가 안 된다.
→ **두 모드로 분리**하여 둘 다 만족한다.

### 표준 사양 (변경 금지 핵심)
| 요소 | 사양 |
|------|------|
| **크게 보기**(기본) | 좌측 썸네일 목록 + 우측 대형 뷰어. 선택 증빙을 A4(`aspect-[210/297]`) **fit-width**로 표시 → 글씨 읽힘 |
| **나란히 비교** | 카드 가로 배치, **`flex-1 min-w-[18rem]`** 로 뷰어 폭에 균등 확대(고정폭 금지). 넘치면 가로 스크롤 |
| **줌** | 50~200%, 25% 단위, 100% 리셋 (크게 보기 전용) |
| **전체화면 토글** | `fixed inset-0` 로 화면 가득 ↔ 접기 |
| **모달 리사이즈** | `resize: both` + min/max 크기 |
| **백드롭 가드** | `onMouseDown`이 백드롭에서 시작한 경우에만 닫음. 리사이즈 드래그가 백드롭에서 끝나도 안 닫힘 (필수) |
| **다운로드** | 개별(`onDownload`) + 일괄(`onDownloadAll`) 훅 |

### 사용법
```jsx
import EvidenceViewer from "@/components/EvidenceViewer";

const [preview, setPreview] = useState(null);
const [pvIdx, setPvIdx] = useState(0);
const openPreview = (files, idx = 0) => { setPreview(files); setPvIdx(idx); };

// 트리거: openPreview(증빙배열, 시작인덱스)
<EvidenceViewer
  open={!!preview}
  onClose={() => setPreview(null)}
  files={preview || []}
  startIndex={pvIdx}
  onDownload={(f) => {/* 개별 다운로드 */}}
  onDownloadAll={() => {/* 일괄 저장 */}}
  renderPage={(file, pg) => /* 실제 PDF·이미지 렌더 (미지정 시 더미 A4 플레이스홀더) */}
/>
```
- `files`: `[{ name, size, pages? }]`
- 실제 증빙 렌더는 `renderPage`로 주입 (Supabase Storage 연동 시 Signed URL → PDF/이미지)

## 2. 결재 사이드 패널 — `components/ApprovalDrawer.jsx` (2026-06-08)

### 적용 대상
상신함·결재함 등에서 통제(행) 클릭 시 우측 슬라이드 패널.

### 표준 사양
| 요소 | 사양 |
|------|------|
| 폭 | **`clamp(32rem, 36vw, 44rem)`** + `max-w-full` — 작아도 32rem, 커도 44rem(초대형 모니터 과확장 방지) |
| 닫기 | ESC · 오버레이 클릭 · X 버튼 |
| 구성 슬롯 | `title` / `subtitle` / 본문(children) / `footer`(고정 액션) |

### 패널 본문 권장 섹션 순서 (증빙 확인 맥락)
통제활동 내용 → 평가 증빙 → 테스트 절차 → 샘플링 정보 → 상신/제출의견 → 첨부 증빙 → 결재현황

- **샘플링 정보**: 기본은 `모집단No.`(=TRANSACTION_ID)·`모집단 내용`만, **마우스 호버 시 상세**(샘플순번·거래일자·참고1·참고2) 표시 (group-hover)
- **첨부 증빙**: 다건 목록(개별 다운로드·미리보기) + 일괄저장 + 전체 미리보기(→ EvidenceViewer)

## 3. 전역 통제기간 셀렉터 — `components/evalTerms.js` (A안, 2026-06-08)

- 헤더에 통제기간(평가기간) 전역 셀렉터 1개. 모든 페이지가 `useEvalTerm()`으로 구독
- `localStorage` + 커스텀이벤트 동기화 (페르소나 토글과 동일 패턴)
- **마감 통제기간**(`closed`)은 조회 전용 → 변경 액션 비활성 + "마감된 통제기간은 조회만 가능합니다"
- ⚠️ 데이터 정합성 규칙은 `.planning/RISK_DECISION_LOG.md` R8·R9 참조 (Supabase 연동 시 필수)

## 4. 공통 목록 화면 패턴 (상신함·결재함 기준)

| 요소 | 표준 |
|------|------|
| 필터 바 | 업무구분·통제번호·결재상태·평가결과 (조회/초기화 버튼) |
| 요약 바 | 전체 N건 · 정상/예외/해당없음 건수 |
| 액션 바 | 좌: 일괄처리 버튼 / 우: 건수표시 → 표시개수(50·100·150·200건 조회) → 엑셀저장 |
| 정렬 | 업무구분 → 통제번호 **자연정렬**(`localeCompare(..., {numeric:true})`) |
| 페이지네이션 | 하단 중앙. 기본 50건 |
| 결재상태 | 미상신 / 진행 / 반려 / 완료 (배지 색: gray / blue / red / emerald) |
| 평가결과 | 정상(emerald) / 예외(red·bold) / 해당없음(gray) |

## 5. 상신·결재 처리 패턴 — 저장/상신 분리 + 2단계 게이트 (2026-06-08 확정)

### 저장과 상신 분리
- **저장**: 데이터만 원본 레코드에 반영 → `상신대기`(DB `DRAFT`). 결재 미시작
- **상신**: 결재 사이클 시작 → `진행`(DB `SUBMITTED`)
- 패널 버튼: `[저장]` / `[저장 후 상신]` 2개

### 재상신 UX = B안 (머문 채 패널 편집)
- 페이지 점프 없음. 상신함/결재함에 머물고 **우측 패널이 편집 모드**로 전환 → 증빙 재업로드·의견 수정 → 데이터는 원본 레코드에 반영
- (Phase 8 결재 UX 확정 "팝업→사이드 패널, 점프 없음"과 일관)

### 상신대기 단일 상태 + 2단계 게이트
- 상태는 **`상신대기` 하나**. "준비완료"는 별도 상태가 아니라 **자동 검사 결과**(파생값). 문제 있을 때만 표시
- **2단계 게이트**:
  | 등급 | 미충족 시 | 우회 |
  |------|----------|------|
  | 🔴 **필수** | 상신 **차단**(버튼 disabled) | 불가 |
  | 🟡 **권장** | 🟡 보완필요 표시 | **사유 입력 후 우회** (사유는 결재자·감사에 노출) |
- 게이트 결과 자동 갱신: 저장 시 체크리스트 재검사 → 모두 충족이면 경고 자동 해제
- 필수/권장 기준은 **P5가 평가설정(v8 M3)에서 통제유형별 지정**

### 일괄상신
- **상신대기 + 깨끗(필수·권장 모두 충족)** 건만 대상. 보완필요 건은 자동 제외 + "개별 상신에서 처리" 안내
- **일괄상신 확인 모달**: 대상 건수 + 결재선 미리보기 + 제외 안내 → 확인 후 실행

### 데이터 (추가 마이그레이션 불필요)
- `evidence_submissions.status`의 기존 `DRAFT/SUBMITTED/APPROVED/REJECTED/REVISION_REQUESTED`로 충족
- 우회 사유·감사추적은 `audit_logs`에 기록 (deviation 문서화 = ICFR 모범관행)
